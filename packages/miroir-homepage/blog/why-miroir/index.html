Why Miroir?
The Philosophy and Rationale

Today's IDEs are paradoxically not that "integrated." Miroir aims to recapture Smalltalk's interactive development experience while addressing its limitations with modern tooling.

May 2026 · 20 min read

The Problem: Fragmented Software Development

The Software Development Quandary

Today's Integrated Development Environments — VS Code, IntelliJ, Eclipse — are paradoxically not-so-much "integrated". They directly address only a fraction of development-related activities:

  • ✅ What IDEs do well: writing code, browsing code, refactoring (to some extent).
  • ⚠️ Limited support through third-party integration: version control, test runners, terminal access.
  • ❌ No support at all: software design and modeling, database schema design and evolution, CI/CD pipelines, runtime monitoring and debugging integration.

The Tool Sprawl Problem

Modern software development requires juggling dozens of disparate tools, each with its own domain-specific language (SQL, GraphQL, YAML, Terraform…), its own mental model, configuration overhead, and learning curve. There is very little interoperability among these languages — even sometimes among DSLs within the same tool.

The Separation of Concerns Paradox

Mainstream IDEs boast about being "unopinionated" — independent from any particular software design method. This fosters wide acceptance. However, using different tools and methods at conception and development time vs. at runtime makes encompassing the whole lifecycle of a software system considerably more difficult.

Software development becomes paradoxically considered separate from the runtime workings of the system itself, barring any holistic approach from a Systems Engineering point of view.

We view this as a major hindrance to improving software development activities themselves. This tendency has been somewhat alleviated by CI/CD methodologies — but only the tools we use still lag in features to address these practices fully.

The Proposal: Integrated Development

Miroir aims to integrate development-time and runtime activities in a unified environment, where:

  • Development, testing, and deployment are not separate phases but continuous activities.
  • Models are executable and evolve with your application.
  • Feedback is immediate — no compile-wait-run-debug cycle.
  • Everything is data — models, queries, transformations, UI definitions are all machine-readable JSON.
  • DSLs are first-class citizens — create, integrate, and evolve domain languages at runtime.

Inspiration from Smalltalk

The Paradise Lost

Looking to the past for inspiration, Smalltalk pioneered a radically different, interactive approach to software development. In Smalltalk, development-time and runtime activities happen in the same environment. On a small scale, development, testing, and deployment become essentially the same activity.

"Having perfect, run-time-precise, immediate feedback during development amounts to many developers to being in a proverbial 'ideal world', a paradise lost which many have endeavoured to reach again."

Accelerating feedback is now widely recognized as a primary means to increase software quality and developer productivity (see Accelerate). Smalltalk achieved this decades ago.

Smalltalk's Limitations

Yet Smalltalk had important limitations — many issues were outside the scope of what any development environment could reach at the time:

  • No support for version control or delivery processes.
  • No direct support for automated testing.
  • No support for refactoring tools.
  • No support for software or database model design.
  • Limited ecosystem and integration with external systems.

Miroir aims to recapture Smalltalk's interactive development experience while addressing its limitations with modern tooling.

The Miroir Vision

In Miroir, your entire application — its data model, business logic, UI reports, and endpoints — is expressed as JSON declarations. These declarations are themselves stored as data in the running system, making them introspectable, versionable, and modifiable at runtime.

This means you can evolve your application while it runs. Add a new field to an entity and watch it propagate to forms, queries, and reports — immediately, without restarting. This is the Smalltalk dream, rebuilt on modern foundations: TypeScript, React, PostgreSQL, and standard web protocols.

Model-Based Software Engineering

Miroir is grounded in model-based software engineering. The "model" is not a UML diagram or a separate artifact — it is the running application's schema. Entities, their relations, validation rules, queries, and UI layouts all live in the same store as your data, bootstrapped and self-describing.

What You Give, What You Get

Miroir asks you to describe your domain in its meta-language (Jzod schemas). In return, you get:

  • Generated TypeScript types and Zod validators — always in sync with your schema.
  • Auto-generated CRUD reports and forms from entity definitions.
  • A composable query language (extractors, combiners, transformers) that runs identically in-browser or on the server.
  • An MCP interface so AI agents can query and modify your model directly.
  • A live design mode for runtime schema evolution — no rebuild required.
  • Multi-backend persistence: swap from IndexedDB to PostgreSQL with a config change.

The investment is front-loaded — you learn the meta-language once — but the returns compound: every new entity automatically benefits from all the infrastructure you've already set up.
